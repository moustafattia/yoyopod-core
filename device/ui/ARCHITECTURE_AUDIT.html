UI Host Worker — Architecture Audit & Refactor Plan

Comprehensive analysis of device/ui/ with proposed clean-architecture refactor and standardized worker ↔ runtime-manager protocol.

1. Executive summary

35

UiScreen variants

The Rust UI worker is a single-threaded, stdin/stdout-driven LVGL frontend that consumes RuntimeSnapshot commands from the runtime manager and emits UiIntent events. It is structurally close to a clean-architecture shape (hardware traits, pure screen-model builders, controller pattern), but suffers from three concentrated debt clusters:

  1. Monolithic state machine — runtime/state_machine.rs (628 LOC) mixes navigation, focus, input routing, intent-payload construction, PTT semantics, and overlay preemption.
  2. Hardcoded layout/style tables in LVGL backend — lvgl/native_backend.rs (2,383 LOC) bakes ~200 match arms of layout coords and RGB styles into code.
  3. Stringly-typed protocol — the dispatch in worker.rs matches on raw message_type strings and intent payloads are unstructured serde_json::Value.
Headline recommendation. Introduce four explicit layers (transport → application → presentation → render) with a typed UiCommand / UiEvent envelope, a declarative navigation graph, and a frame-clocked transition layer. The current crate already has the right files; what is missing is a stable boundary between them.
high severity medium low / nit

3.2 What is already clean

  • hardware/ traits cleanly separate device I/O from the worker.
  • screens/*.rs builder functions are pure: (&RuntimeSnapshot, focus) → ScreenModel. Zero LVGL leakage.
  • screens/models.rs view-models are render-engine-agnostic.
  • ScreenController trait isolates LVGL imperative code per screen.
  • Single envelope type WorkerEnvelope on the wire (no ad-hoc per-message structs).

3.3 What violates the boundaries

  • worker.rs simultaneously plays transport, dispatcher, render-tick driver, input poller, and intent emitter.
  • state_machine.rs bakes navigation graph, focus rules, intent json!{} payload-building, and PTT semantics into one file.
  • native_backend.rs hardcodes per-role layout coords and RGB styles instead of consuming a table.
  • UiScreen, NativeSceneKey, ScreenModel are three parallel enums that must stay synchronized by hand.
  • UiIntent payloads are untyped serde_json::Value.

4. Wire protocol surface

4.1 Envelope (verbatim)

Defined in device/protocol/src/lib.rs:

pub struct WorkerEnvelope {
    pub schema_version: u16,         // SUPPORTED_SCHEMA_VERSION = 1
    pub kind: EnvelopeKind,          // Command | Event | Result | Error | Heartbeat
    pub message_type: String,        // "ui.tick", "ui.intent", ...
    pub request_id: Option<String>,
    pub timestamp_ms: u64,
    pub deadline_ms: u64,
    pub payload: Value,              // must be object, not null
}

Protocol gaps.
  • No schema version negotiation beyond the global schema_version field.
  • No delta updates — entire RuntimeSnapshot (5–50 KB) re-sent every tick.
  • No ui.shutdown_complete ack — runtime manager cannot confirm clean teardown.
  • Intent payloads are Value: no compile-time contract between worker and runtime manager.
  • PTT press/release are consumed internally and never surface to the runtime manager.
  • Unknown message_type is silently dropped (continue in worker loop) — no ui.error emitted.

6. Threading, lifecycle & state

6.1 Threading

  • Single OS thread. Blocking stdin read drives everything. No tokio, no futures, no channels.
  • Button read (~50 ns), LVGL render and SPI flush (~11 ms on Whisplay) are synchronous on the worker thread.
  • LVGL's flush callback re-enters Rust synchronously inside renderer.render() and pushes pixels to the display device.

6.2 Startup ordering

  1. main.rs parses CLI, instantiates MockDisplay | WhisplayDisplay + button.
  2. run_worker constructs an empty UiRuntime with default RuntimeSnapshot.
  3. If native-lvgl: LvglRenderer::open() initializes LVGL display, draw buffer, flush callback.
  4. Emit ui.ready { display }.
  5. Enter blocking message loop.

6.3 Shutdown

  • On ui.shutdown / EOF, the loop breaks. Drops cascade: LvglRenderer → RustSceneBridge → active ScreenController::teardown() → framebuffer → display → button.
  • No signal handler: SIGTERM kills the process without controller teardown — GPIO state may persist.
  • No ui.shutdown_complete sent.

10. Dead code, TODOs, naming inconsistencies

10.1 Dead / vestigial

  • UiRuntime.last_app_state — written, never read.
  • LvglRenderer::open(explicit_source) — parameter ignored; LVGL_SOURCE_DIR env var wins.
  • PowerPageSnapshot.icon_key — never rendered by controllers/power.rs.
  • VoiceNoteSummarySnapshot.unread — displayed but never cleared.
  • OneButtonMachine.hold_back_fired — defensive flag; current caller doesn't need it.
  • hardware/mock.rs hardcoded for 240×280 — undocumented coupling to Whisplay size.

10.2 Naming inconsistencies

  • Scene vs Screen — NativeSceneKey::Hub and UiScreen::Hub mean the same thing in different layers.
  • Singular vs plural — UiScreen::Playlist ↔ ScreenModel::Playlists.
  • Builder naming — hub_model / listen_model / playlists_model / now_playing_model have no pattern.
  • Feature flag casing — native-lvgl (hyphen in Cargo.toml) vs cfg(feature = "native-lvgl") — easy to mistype as underscore.

11. Error handling & safety gaps

  • High — LVGL FFI assumes non-null return from lv_obj_create; null = panic, no recovery.
  • Med — Unknown app_state string silently falls back to current screen (state_machine.rs:167).
  • Med — Unknown message_type dropped silently in worker loop.
  • Med — Display flush return values ignored in some paths.
  • Med — No watchdog: if runtime manager dies, worker keeps rendering the last snapshot forever.
  • Low — Inconsistent anyhow::Result vs typed Result across crate boundaries.

12.3 Core design moves

  1. Typed envelope — replace string dispatch with an exhaustive UiCommand / UiEvent enum, serde-tagged on message_type. Compiler-checked.
  2. Typed UiIntent — kill serde_json::Value payloads. Each variant has its own struct. Shared with the runtime manager via device/protocol.
  3. Declarative navigation graph — replace select_focused match arms with a HashMap<UiScreen, ScreenSpec> table loaded once.
  4. Single screen identity — collapse UiScreen, NativeSceneKey, ScreenModel into one enum + a trait Screen with associated ViewModel + Controller types.
  5. Data-driven layout/theme — extract the 475-LOC layout_for_role match and the 75-LOC style match into layout.ron / theme.ron loaded at startup. native_backend.rs drops to ~600 LOC of FFI only.
  6. Transitions/animation layer — explicit struct Transition { target, prop, from, to, duration_ms, started_at, easing }. The render loop ticks transitions and the view model returns interpolated values — controllers stay stateless.
  7. Snapshot deltas — extend protocol with ui.runtime_patch { path, value } alongside ui.runtime_snapshot for partial updates.
  8. Per-screen Controller<Model = T> trait — typed associated Model; eliminates the runtime let ScreenModel::Hub(_) = … pattern.

14. Standardized envelope & typed messages

14.1 Wire stays compatible

WorkerEnvelope is unchanged on the wire. The change is purely inside the worker and the runtime manager: both decode/encode into typed enums shared via device/protocol.

14.3 What this buys us

  • Exhaustiveness checking — the compiler enforces every command has a handler.
  • Single source of truth — the runtime manager and worker share the same enums; no parallel JSON contracts.
  • Discoverable surface — cargo doc renders the protocol.
  • Adding a new command/intent = one variant + one handler — no string typos.
  • Backwards-compatible wire — schema is unchanged; rolling deploys are safe.

16. Worked example: adding animation

Today (current code)

Adding "fade-in the now-playing screen over 300 ms" touches ~15 files: a new string in worker dispatch, a new field in UiRuntime, new LvglFacade methods, every controller's sync() checking whether each widget is animating, a render-loop tweak to keep marking dirty, and lifecycle hooks to cancel on scene teardown.

After refactor

  1. UiCommand::Animate(AnimationRequest) — one enum variant.
  2. AppCore::on_event(UiCommand::Animate(req)) → state.transitions.push(req.into()) — one match arm.
  3. Each tick, presentation/transitions.rs advances all transitions and the view model returns interpolated values.
  4. Controllers stay completely unchanged — they just call facade.set_accent(widget, view_model.signal_color); the color happens to be interpolated.
// presentation/transitions.rs
pub struct Transition<V> {
    pub target_role: WidgetRole,
    pub prop: AnimatableProp,
    pub from: V,
    pub to: V,
    pub started_at_ms: u64,
    pub duration_ms: u32,
    pub easing: Easing,
}

impl<V: Lerp> Transition<V> {
    pub fn value_at(&self, now_ms: u64) -> V {
        let t = ((now_ms - self.started_at_ms) as f32 / self.duration_ms as f32).min(1.0);
        V::lerp(&self.from, &self.to, self.easing.apply(t))
    }
}

Adding new animatable properties = one variant of AnimatableProp. Adding new screens that animate = nothing — they get it for free.
